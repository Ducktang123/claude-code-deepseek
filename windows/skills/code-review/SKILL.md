# 代码审查

对目标代码做一次务实的审查，**只报有把握的问题**，不要堆砌泛泛的建议。

## 步骤
1. 确定审查范围：
   - 如果在 git 仓库且用户没指定文件，先看改动：`git diff`（未提交）或 `git diff HEAD~1`。
   - 否则审查用户指定的文件。
2. 逐项检查并分类：
   - **Bug / 正确性**：逻辑错误、边界条件、空值、并发、资源泄漏。
   - **安全**：注入、越权、硬编码密钥、不安全的反序列化。
   - **可简化 / 可复用**：重复代码、过度复杂、可用现成库或已有函数替代。
   - **效率**：明显的性能问题（N+1、不必要的循环/拷贝）。
3. 输出格式（中文）：每条给出 `文件:行号`、问题、为什么是问题、建议改法。
4. 按严重程度排序：先严重 bug，再安全，再清理项。

## 原则
- 没问题就说"没发现明显问题"，不要硬找。
- 风格类（命名/缩进）除非影响可读性，否则少提。
- 改法要贴合现有代码风格。
